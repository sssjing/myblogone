                              # 什么是 Function Calling

                              Function Calling（函数调用）是大语言模型（LLM）与外部世界交互的核心机制。它允许模型在生成回答的过程中，按需调用外部工具、API 或函数，从而突破纯文本生成的局限，获取实时数据、执行计算、操作数据库等。

                              OpenAI 在 2023 年 6 月首次引入了 Function Calling 功能，使得 GPT 模型能够以结构化的 JSON 格式输出函数名和参数，由外部系统执行后再将结果返回给模型。此后，几乎所有主流 LLM 提供商都支持了类似的能力，包括 Anthropic 的 Tool Use、Google 的 Function Calling，以及开源模型社区的各种实现。

                              Function Calling 的本质，是让 LLM 从一个"只能说话"的对话者，变成一个"能做事"的智能体。

                              # Function Calling 的工作流程

                              根据李宏毅老师的笔记，Function Calling 的完整工作流程可以分为以下几个步骤：

                              1. 定义工具：在 System Prompt 中定义所有可用的工具，包括工具名称、功能描述、参数类型和约束条件。
                              2. 用户发送消息：用户向模型发送一个请求，例如"今天北京的天气怎么样？"。
                              3. 模型决策：LLM 分析用户意图，判断是否需要调用工具。如果需要，它不会直接生成自然语言回答，而是输出一个结构化的工具调用请求。
                              4. 输出工具调用：模型输出 JSON 格式的工具调用，包含函数名和参数。例如：{"name": "get_weather", "arguments": {"city": "北京"}}。
                              5. 系统执行工具：应用程序接收工具调用请求，执行对应的函数（如调用天气 API），获取结果。
                              6. 结果回传：将工具执行的结果作为新的上下文信息，反馈给 LLM。
                              7. 生成最终回答：LLM 基于工具返回的结果，生成用户可读的自然语言回答："今天北京天气晴朗，气温 25°C，适合外出。"

                              这个流程可以用一个简单的伪代码来理解：

                              # 伪代码：Function Calling 流程
tools = [
    {
        "name": "get_weather",
        "description": "获取指定城市的天气信息",
        "parameters": {
            "city": {"type": "string", "description": "城市名称"}
        }
    }
]

# Step 1: 发送消息给 LLM
response = llm.chat(messages, tools=tools)

# Step 2: 检查是否需要调用工具
if response.tool_calls:
    tool_call = response.tool_calls[0]
    # Step 3: 执行工具
    result = execute_tool(tool_call.name, tool_call.arguments)
    # Step 4: 将结果回传给 LLM
    final_response = llm.chat(messages + tool_result)

                              # 工具描述的重要性

                              在 Function Calling 中，工具描述（Tool Description）的质量直接决定了模型能否正确地选择和使用工具。可以说，工具描述就是面向 LLM 的 Prompt Engineering。

                              # 好的工具描述

                              • 清晰具体：准确描述工具的功能和适用场景
                              • 参数约束明确：说明每个参数的类型、格式、取值范围
                              • 包含示例：提供典型的调用示例，帮助模型理解用法
                              • 边界条件：说明工具不能做什么，避免误用

                              # 差的工具描述

                              • 模糊不清：例如"获取信息"——获取什么信息？从哪里获取？
                              • 缺少参数说明：参数含义不明，模型容易传入错误的值
                              • 描述与实现不一致：描述说返回 JSON，实际返回纯文本

                              一个好的经验法则是：如果你自己看完工具描述后都不知道该怎么用，那 LLM 也不可能用好它。

                              # 工具太多怎么办

                              在实际应用中，一个 Agent 可能需要访问数百甚至上千个工具。这时候问题就来了：上下文窗口（Context Window）装不下所有的工具描述。

                              根据李宏毅老师的笔记，当工具数量过多时，常见的解决方案包括：

                              # 基于 RAG 的工具检索

                              将所有工具的描述存储到向量数据库中，根据用户的任务动态检索最相关的 Top-K 个工具：

                              1. 将每个工具的名称和描述转换为向量嵌入（Embedding）
                              2. 存储到向量数据库（如 FAISS、Pinecone）
                              3. 用户请求到来时，将任务描述也转换为向量
                              4. 通过相似度检索，找到最相关的 K 个工具
                              5. 只将这 K 个工具的描述放入上下文中

                              # 工具分层与路由

                              将工具按照功能域进行分类，先由一个"路由"模型判断任务属于哪个域，再加载该域下的工具集。例如：

                              • 天气域：get_weather, get_forecast, get_air_quality
                              • 邮件域：send_email, read_email, search_email
                              • 日历域：create_event, list_events, update_event

                              # 相关基准测试

                              学术界已经有一些针对大规模工具使用的基准测试：

                              • ToolBench：包含 16000+ 真实 API，测试模型在大规模工具集下的表现
                              • API-Bank：评估模型调用 API 的准确性、参数填充的正确性

                              # 工具结果有误怎么办

                              一个常见的误区是认为"工具返回的结果一定是对的"。实际上，模型在处理工具返回结果时面临一个棘手的问题：过度信任。

                              # 过度信任问题

                              研究表明，LLM 倾向于过度信任工具返回的结果，尤其是当工具结果与模型内部知识一致时。更危险的是，即使工具结果与模型的内部知识明显矛盾，模型仍然可能选择信任工具。

                              例如，如果天气 API 返回了一个明显错误的数据（如"北京今天的温度是 200°C"），模型可能仍然会将其包装成一个看起来合理的回答，而不是质疑结果的正确性。

                              # 解决方案

                              • 校准机制：在 Prompt 中明确要求模型对工具结果进行合理性检查
                              • 置信度评分：让模型对工具结果的可信度进行打分，低于阈值时触发人工审核
                              • 多源交叉验证：调用多个独立的工具，对比结果是否一致
                              • Anthropic 的方法：允许模型在回答中表达对工具结果的不确定性，例如"根据工具返回的数据，但我不完全确定这个结果的准确性"

                              # MCP (Model Context Protocol)

                              MCP（模型上下文协议）是 Anthropic 推出的一项标准化协议，旨在为 LLM 与外部工具之间的交互提供统一的接口规范。

                              # 为什么需要 MCP

                              在 MCP 出现之前，每个 LLM 提供商都有自己的工具调用格式，每个应用都需要为不同的模型编写适配代码。这就像早期互联网时代，每个网站都有自己的数据格式，直到 HTTP 和 HTML 的出现才统一了 Web 通信。

                              MCP 的目标就是成为 AI 领域的"HTTP"——一个通用的协议标准。

                              # MCP 的核心概念

                              • Tools（工具）：模型可以调用的函数，有明确的输入输出 schema
                              • Resources（资源）：模型可以读取的数据源，如文件、数据库
                              • Prompts（提示模板）：预定义的提示词模板，用于特定任务

                              # MCP 生态

                              MCP 正在快速发展，已经形成了一个不断增长的生态系统。开发者可以使用 MCP SDK（支持 TypeScript、Python 等语言）快速构建 MCP Server，将现有的 API 和工具暴露给 LLM 使用。越来越多的 IDE、开发工具和应用开始原生支持 MCP。

                              # 常见的工具类型

                              在 AI Agent 的实际应用中，常见的工具可以分为以下几大类：

                              # 搜索类

                              • 网络搜索：通过搜索引擎获取实时互联网信息
                              • 知识库检索：从内部文档、Wiki、FAQ 中检索相关信息
                              • 向量搜索：基于语义相似度的检索，适用于非结构化数据

                              # 代码类

                              • 代码执行：在沙箱中运行 Python、JavaScript 等代码
                              • 文件操作：读取、写入、编辑文件
                              • 代码分析：静态分析、lint 检查、代码搜索

                              # 数据类

                              • 数据库查询：执行 SQL 查询，获取结构化数据
                              • API 调用：调用第三方 REST/GraphQL API
                              • 数据分析：数据统计、可视化、报表生成

                              # 通信类

                              • 邮件：发送、接收、搜索邮件
                              • 即时通讯：发送消息到 Slack、Teams 等平台
                              • 通知：发送推送通知、短信

                              # 媒体类

                              • 图像生成：通过 DALL-E、Midjourney 等生成图像
                              • 音频处理：语音识别（STT）、语音合成（TTS）
                              • 视频处理：视频剪辑、字幕生成
