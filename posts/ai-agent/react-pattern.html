ReAct 模式与 Agent 架构

杨景2025-05-27大约 10 分钟

# 从 Chain-of-Thought 到 ReAct

在深入 ReAct 模式之前，我们需要先理解它的前身——Chain-of-Thought（CoT）。

2022 年，Wei 等人在论文 "Chain-of-Thought Prompting Elicits Reasoning in Large Language Models" 中提出了 CoT 方法。其核心思想非常简单：通过在 Prompt 中加入 "Let's think step by step" 这样的引导语，让大语言模型在输出最终答案之前，先展示中间推理过程。这种方法显著提升了模型在数学推理、逻辑推理等任务上的表现。

CoT 的局限性：CoT 是一种纯粹的推理（Reasoning）方法，模型只能在"脑内"进行思考，无法与外部环境交互。当任务需要查询实时信息、调用外部工具或根据环境反馈调整策略时，CoT 就显得力不从心。

2022 年，Yao 等人在论文 "ReAct: Synergizing Reasoning and Acting in Language Models" 中提出了 ReAct 模式。ReAct 将 Reasoning（推理）和 Acting（行动）结合在一个统一的循环中，让模型不仅能"想"，还能"做"，并根据行动的结果继续思考。这一范式成为了现代 AI Agent 架构的基石。

# ReAct 模式详解

# 核心循环：Thought → Action → Observation

ReAct 的核心是一个简洁而强大的循环，由三个步骤组成：

  1. Thought（思考）：LLM 对当前状态进行推理，分析下一步应该做什么。这一步是模型的"内心独白"，帮助模型分解复杂问题、制定策略。
  2. Action（行动）：LLM 根据思考的结果，选择一个具体的行动。行动可以是调用搜索引擎、查询数据库、执行代码、调用 API 等任何可用的工具。
  3. Observation（观察）：环境（工具）返回行动的结果。这个结果会被反馈给 LLM，作为下一轮思考的输入。

这个循环会不断重复，直到任务完成或达到终止条件。

# 示例：回答一个问题

假设用户提问："李宏毅的 Agent 课程是什么时候发布的？"

ReAct 模式下的执行过程如下：

Question: 李宏毅的 Agent 课程是什么时候发布的？

Thought: 我需要查找李宏毅 Agent 课程的发布时间。这是一个需要外部信息的问题，我应该使用搜索工具。
Action: search("李宏毅 Agent 课程 2025")
Observation: 搜索结果显示：李宏毅教授的 AI Agent 课程于 2025 年春季学期在台湾大学开设，课程视频同步发布在 YouTube 上。

Thought: 根据搜索结果，课程是 2025 年春季发布的。我现在可以回答用户的问题了。
Action: finish("李宏毅的 Agent 课程于 2025 年春季发布。")

从这个例子可以清晰地看到，Thought 让模型能够有条理地分解问题，Action 让模型能够获取外部信息，Observation 则为下一轮推理提供了新的上下文。三者协同工作，使得模型能够解决那些仅靠内部知识无法回答的问题。

# ReAct 相比 CoT 的优势

  • 可解释性更强：Thought 步骤显式地展示了模型的推理过程，便于调试和审计。
  • 能够获取外部信息：通过 Action 步骤调用工具，模型可以访问实时数据、专业知识库等外部资源。
  • 可纠错：当 Observation 显示行动结果不符合预期时，模型可以在下一轮 Thought 中调整策略。
  • 减少幻觉：通过与真实环境交互获取信息，而不是仅凭模型内部知识"编造"答案。

# Anthropic 的五大工作流模式

Anthropic 在其技术文档 "Building Effective Agents" 中总结了五种常见的 Agent 工作流模式。这些模式从简单到复杂，适用于不同场景。

# 1. Prompt Chaining（提示链）

将复杂任务分解为多个顺序执行的子任务，每个子任务的输出作为下一个子任务的输入。

任务 → LLM1（生成草稿）→ 质量检查 → LLM2（润色优化）→ 最终输出

适用场景：文档生成、多步骤翻译、需要中间验证的复杂任务。关键在于每一步的输出质量都可以被检查，不合格时可以重试或回退。

# 2. Routing（路由分发）

根据输入的类型或内容，将请求路由到不同的专门处理流程。

输入 → 分类器 LLM → [技术问题] → 技术支持流程
                    → [账单问题] → 账单处理流程
                    → [一般咨询] → 通用回复流程

适用场景：客服系统、工单分流、内容分类。通过路由可以为不同类型的问题提供专门优化的 Prompt 和处理逻辑。

# 3. Parallelization（并行化）

同时运行多个 LLM 调用，然后聚合结果。

输入 → LLM1 ──┐
       LLM2 ──┼→ 聚合器 → 输出
       LLM3 ──┘

适用场景：多角度评估、投票决策、批量处理。例如让多个模型同时评估一段代码的质量，然后取共识结果。

# 4. Orchestrator-Workers（编排者-工作者）

一个中心 LLM 充当编排者（Orchestrator），负责分解任务并协调多个工作者 LLM（Workers）完成子任务。

复杂任务 → 编排者 LLM → 子任务1 → Worker LLM1 → 结果1
                       → 子任务2 → Worker LLM2 → 结果2
                       → 子任务3 → Worker LLM3 → 结果3
                       → 汇总 → 最终输出

适用场景：复杂的研究任务、大型代码重构、多文档分析。编排者动态决定需要哪些子任务以及如何分配。

# 5. Evaluator-Optimizer（评估者-优化者）

一个 LLM 生成内容，另一个 LLM 评估质量，形成"生成→评估→优化"的迭代循环。

需求 → 生成器 LLM → 输出 → 评估器 LLM → [通过] → 最终输出
                                       → [不通过] → 反馈 → 生成器 LLM → ...

适用场景：代码生成、写作优化、翻译润色。当任务有明确的质量标准时，这种模式尤为有效。

# Agent 循环的终止条件

ReAct 循环不能无限运行下去，合理的终止条件是 Agent 系统设计的重要一环。常见的终止条件包括：

  • 任务完成：Agent 通过 finish() 动作明确表示任务已完成，并返回最终答案。这是最理想的终止方式。
  • 达到最大迭代次数：设置一个硬性的上限（如 10 轮或 20 轮），防止 Agent 陷入无限循环。这是一种安全保护机制。
  • 错误或置信度过低：当 Agent 连续多次行动失败，或模型对当前状态的置信度低于阈值时，主动终止并返回错误信息。
  • 用户干预：在交互式场景中，用户可以在任意时刻中断 Agent 的执行，查看中间状态并手动调整方向。
  • 超时：设置时间限制，避免 Agent 在某个步骤上耗时过长。

在实际系统中，通常会同时配置多个终止条件，形成多层安全网。例如：优先检查任务是否完成，其次检查是否出错，最后检查是否超时或超过最大迭代次数。

# ReAct vs Function Calling

很多人会混淆 ReAct 和 Function Calling，实际上它们是不同层面的概念：

  • ReAct 是一种概念模式（Conceptual Pattern），描述的是"推理-行动-观察"的循环范式。它关注的是 Agent 如何思考和决策。
  • Function Calling 是一种实现机制（Implementation Mechanism），由 OpenAI、Anthropic 等厂商提供的 API 能力，允许模型以结构化的 JSON 格式调用预定义的函数。它关注的是模型如何与外部工具对接。

在现代 LLM 系统中，两者通常是结合使用的：

  1. 模型在文本中进行 Reasoning（对应 ReAct 的 Thought）
  2. 通过 Function Calling 机制发起结构化的工具调用（对应 ReAct 的 Action）
  3. 将工具返回的结果注入到上下文中（对应 ReAct 的 Observation）
  4. 模型基于新信息继续推理（回到步骤 1）

可以说，ReAct 提供了 Agent 的"思维方式"，而 Function Calling 提供了 Agent 的"行动手段"。两者缺一不可。
