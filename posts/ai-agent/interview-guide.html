AI Agent 面经与面试题

杨景2025-05-27大约 15 分钟

本文整理了 AI Agent 方向的高频面试题与面经，涵盖基础概念、RAG、Function Calling、Multi-Agent 以及高级话题等核心知识点，适合准备 AI Agent 相关岗位面试的同学参考复习。

# 基础概念类

# Q: 什么是 AI Agent？和 Chatbot 有什么区别？

AI Agent 是以大语言模型（LLM）为核心推理引擎，结合 Planning（规划）、Memory（记忆） 和 Tool Use（工具使用） 三大模块，能够自主感知环境、制定计划、执行动作并根据反馈迭代优化的智能体系统。其核心公式可以表示为：Agent = LLM + Planning + Memory + Tool Use。

与传统 Chatbot 相比，二者存在本质差异。Chatbot 本质上是 被动式（reactive） 的对话系统，无论是单轮还是多轮对话，都只能根据用户输入做出回应，缺乏自主性。而 Agent 是 主动式（proactive） 的，它能够自主规划任务步骤、调用外部工具（如搜索引擎、代码执行器、数据库等）、利用记忆系统保持上下文连贯性，并在遇到错误时自动调整策略重试。最关键的区别在于 自主性（autonomy） 和 环境交互能力：Agent 能够独立完成复杂的多步骤任务，而不仅仅是回答问题。

面试要点：回答时要突出 Agent 的四大核心能力（规划、记忆、工具、反思），并用具体场景举例说明 Agent 相比 Chatbot 的优势。

# Q: Agent 的核心架构是什么？

目前业界最广泛引用的 Agent 架构来自 Lilian Weng 的博客文章 "LLM Powered Autonomous Agents"。该架构将 LLM 作为 Agent 的"大脑"，围绕其构建三大核心模块：

  1. Planning（规划）：包括任务分解（Task Decomposition）和自我反思（Self-Reflection）。任务分解将复杂目标拆解为可执行的子任务序列；自我反思则允许 Agent 在执行后评估结果并调整后续计划。常见技术包括 Chain of Thought（CoT）、Tree of Thoughts（ToT）以及 Reflexion 等。
  2. Memory（记忆）：分为短期记忆和长期记忆。短期记忆对应模型的上下文窗口（Context Window），用于维持当前对话的状态；长期记忆通常借助外部向量数据库（Vector Store）实现，可以存储和检索历史交互中的关键信息。
  3. Tool Use（工具使用）：通过 Function Calling 等机制，Agent 可以调用外部 API、代码解释器、数据库查询等工具来获取信息或执行操作，从而突破 LLM 自身知识和能力的边界。

面试要点：画图说明三大模块与 LLM 之间的数据流关系，强调这是一个循环迭代的系统而非线性流程。

# Q: 什么是 ReAct 模式？

ReAct（Reasoning + Acting） 是一种将推理与行动交替进行的 Agent 执行范式，由 Yao et al. 在 2022 年提出。其核心循环为：Thought（思考） → Action（行动） → Observation（观察）。

在每一轮循环中，Agent 首先通过 Thought 进行推理分析当前状态和下一步计划；然后通过 Action 执行具体操作（如搜索、计算、调用 API）；最后通过 Observation 获取行动的结果，并将其作为下一轮推理的输入。这种模式的优势在于：

  • 相比纯 Chain of Thought（仅有推理没有行动），ReAct 能够获取外部信息，避免"幻觉"；
  • 相比纯 Acting（仅有行动没有推理），ReAct 的推理步骤使得决策过程更加可解释、更不容易出错；
  • 推理与行动的交替使得 Agent 能够根据中间结果动态调整策略。

面试要点：最好能手写一个简单的 ReAct 循环伪代码，展示 Thought/Action/Observation 的交替过程。

# RAG 相关

# Q: 什么是 RAG？和 Fine-tuning 如何选择？

RAG（Retrieval-Augmented Generation，检索增强生成） 是一种在生成回答之前先从外部知识库中检索相关文档，然后将检索到的上下文信息注入到 LLM 的 prompt 中辅助生成的技术方案。其核心思想是"先查后答"，让模型基于真实数据而非仅凭参数记忆来回答问题。

Fine-tuning（微调） 则是使用领域特定数据对预训练模型的参数进行调整，使模型习得特定的知识或风格。

选择策略如下：

  • 选择 RAG 的场景：知识频繁更新（如新闻、产品信息）、需要引用来源以保证可追溯性、计算资源有限无法微调大模型、对数据隐私有要求（知识库可独立管理）。
  • 选择 Fine-tuning 的场景：领域知识相对稳定、需要改变模型的输出风格或格式、对响应延迟有严格要求（RAG 需要额外的检索步骤）、需要提升模型在特定任务上的基础能力。
  • 实践中两者常结合使用：通过轻量级微调提升模型对特定领域文档的理解能力，同时保留 RAG 架构以支持动态知识更新。

面试要点：强调"没有银弹"，需要根据业务场景的具体需求来选择，实际项目中两者经常组合使用。

# Q: RAG 的核心流程是什么？

一个标准的 RAG 系统包含以下核心流程：

  1. 文档加载（Document Loading）：从各种数据源（PDF、网页、数据库等）加载原始文档。
  2. 文本分块（Chunking）：将长文档切分为适当大小的文本块。关键决策包括分块大小（通常 256-1024 tokens）、重叠比例（通常 10%-20%）以及分块策略（固定长度、语义分块、递归分块等）。
  3. 向量化（Embedding）：使用 Embedding 模型（如 OpenAI text-embedding-3、BGE、GTE 等）将文本块转换为高维向量表示。
  4. 向量存储（Vector Store）：将向量存入向量数据库（如 FAISS、Milvus、Pinecone、Chroma 等），建立索引以支持高效检索。
  5. 检索（Retrieval）：根据用户查询检索最相关的文本块。常见策略包括相似度检索（Cosine Similarity）、最大边际相关（MMR，平衡相关性与多样性）以及混合检索（Hybrid Search，结合关键词与语义检索）。
  6. 生成（Generation）：将检索到的文档片段与用户问题组合成 prompt，送入 LLM 生成最终回答。

面试要点：重点说明每个步骤的关键决策点和 trade-off，尤其是 Chunking 策略和 Retrieval 策略的选择。

# Q: 如何评估 RAG 系统的效果？

RAG 系统的评估需要从检索质量和生成质量两个维度进行：

检索评估指标：

  • Recall@K：前 K 个检索结果中包含正确文档的比例，衡量检索的覆盖率。
  • MRR（Mean Reciprocal Rank）：正确文档在检索结果中排名的倒数的平均值，衡量排序质量。
  • NDCG（Normalized Discounted Cumulative Gain）：考虑位置权重的排序质量指标。

生成评估指标：

  • Faithfulness（忠实度）：生成的回答是否忠实于检索到的上下文，是否存在"幻觉"。
  • Answer Relevancy（回答相关性）：生成的回答是否与用户问题相关。
  • Context Relevancy（上下文相关性）：检索到的上下文是否与用户问题相关。

常用评估框架：RAGAS、TruLens、DeepEval 等开源框架提供了端到端的 RAG 评估能力，支持自动化评估和人工评估相结合。

面试要点：能够区分检索端和生成端的评估指标，并说明为什么需要分别评估（检索好不等于生成好）。

# Function Calling 相关

# Q: Function Calling 的工作原理是什么？

Function Calling 是让 LLM 能够结构化地调用外部工具的机制，其工作流程如下：

  1. 工具定义：在 System Prompt 中以 JSON Schema 格式定义可用的工具列表，包括工具名称、功能描述和参数规格。
  2. 决策与输出：当 LLM 判断需要使用工具时，不会直接执行函数，而是生成一个结构化的 JSON 输出，包含要调用的函数名和参数。
  3. 系统执行：应用层解析 LLM 的结构化输出，在实际环境中执行对应的函数调用。
  4. 结果回传：将函数执行结果以文本形式反馈给 LLM，LLM 基于结果继续生成最终回答。

需要特别注意的是，LLM 本身并不执行任何函数，它只是根据工具描述生成结构化的调用意图。真正的函数执行发生在应用层，这保证了安全性和可控性。

面试要点：强调 LLM 只是"生成调用意图"而非"执行函数"，这是面试官经常追问的点。

# Q: 如何处理工具调用失败的情况？

工具调用失败是 Agent 系统中必须妥善处理的场景，常见策略包括：

  • 重试与退避（Retry with Backoff）：对于临时性错误（如网络超时），采用指数退避策略进行重试，避免频繁调用导致服务降级。
  • 降级到备选工具（Fallback）：当主工具不可用时，自动切换到功能相似的备选工具。例如搜索引擎 A 不可用时切换到搜索引擎 B。
  • 通知用户并请求指导：当 Agent 无法自行恢复时，向用户清晰地报告错误信息并请求进一步指示。
  • 优雅降级（Graceful Degradation）：在工具完全不可用的情况下，Agent 基于自身知识尝试直接回答，并明确告知用户回答可能不够准确。

面试要点：体现"防御性编程"思维，说明在生产环境中工具调用的可靠性设计至关重要。

# Multi-Agent 相关

# Q: Multi-Agent 系统的优缺点？

优点：

  • 专业化分工：每个 Agent 专注特定领域，可以使用不同的 prompt、工具和模型，实现专业化。
  • 并行执行：独立的子任务可以由不同 Agent 并行处理，提升整体效率。
  • 模块化与可扩展性：系统可以灵活地添加或移除 Agent，便于维护和扩展。
  • 容错能力：单个 Agent 的失败不会导致整个系统崩溃。

缺点：

  • 协调开销：Agent 之间的通信和任务分配需要额外的编排机制，增加了系统复杂度。
  • 通信成本：Agent 之间的信息传递会消耗额外的 token，增加延迟和成本。
  • 调试困难：分布式执行使得问题定位和调试更加复杂。
  • 一致性挑战：多个 Agent 可能产生冲突的决策，需要冲突解决机制。

面试要点：能结合具体框架（如 AutoGen、CrewAI、LangGraph）说明 Multi-Agent 的实际应用场景和编排模式。

# Q: 如何防止 Agent 陷入无限循环？

Agent 陷入无限循环是生产环境中常见的问题，防御措施包括：

  • 最大迭代限制（Max Iterations）：设置硬性的步骤数上限，超过限制后强制终止并返回当前最佳结果。
  • 循环检测（Loop Detection）：记录 Agent 的历史动作序列，当检测到重复的行动模式时主动中断。
  • 置信度阈值（Confidence Threshold）：当 Agent 在多轮迭代后仍无法达到预期置信度时，触发终止或转交人工处理。
  • Human-in-the-Loop 检查点：在关键决策节点引入人工审核，防止 Agent 在错误方向上持续消耗资源。
  • Reflexion 式自我评估：让 Agent 定期反思当前进展，如果判断无法完成任务则主动停止。

面试要点：说明防御机制应该是多层次的，结合硬限制和软判断共同保障系统稳定性。

# 高级话题

# Q: 如何评估 Agent 的效果？

Agent 的评估比单纯的 LLM 评估更加复杂，需要综合多个维度：

主流 Benchmark：

  • AgentBench：覆盖 8 个不同环境的综合 Agent 能力评测，包括操作系统、数据库、知识图谱等场景。
  • WebArena：基于真实网站的 Agent 评测，测试 Agent 在网页环境中完成复杂任务的能力。
  • SWE-bench：评估 Agent 解决真实 GitHub Issue 的能力，是代码 Agent 的重要评测基准。
  • GAIA：通用 AI 助手评测基准，测试 Agent 在多步推理和工具使用方面的综合能力。

核心评估指标：任务完成率（Task Completion Rate）、执行效率（步骤数/时间）、成本消耗（Token 费用）、安全性（是否有有害行为）。目前 人工评估仍然是金标准，自动化评估与人工评估相结合是最佳实践。

面试要点：至少熟悉 2-3 个 Benchmark，能说明每个 Benchmark 侧重评估的能力维度。

# Q: Agent 的安全性问题有哪些？

Agent 系统面临的安全风险远超传统 LLM 应用，主要包括：

  • Prompt 注入（Prompt Injection）：恶意用户通过精心构造的输入劫持 Agent 行为，使其执行非预期操作。例如在检索到的文档中嵌入恶意指令。
  • 工具滥用（Tool Misuse）：Agent 可能被诱导以有害的参数调用工具，如执行危险的系统命令或发送恶意请求。
  • 数据泄露（Data Leakage）：Agent 在处理过程中可能无意间暴露敏感信息，如将内部数据包含在对外的 API 调用中。
  • 资源耗尽：无限循环或大量无效调用可能导致计算资源和 API 配额的浪费。

缓解措施：沙箱隔离（Sandboxing）、最小权限原则（Permission Systems）、输出验证（Output Validation）、速率限制（Rate Limiting）以及人工审核机制。

面试要点：安全性是 Agent 面试中的高频话题，要能从攻击面和防御面两个角度系统地阐述。

# Q: LLM 的规划能力如何？有什么局限？

正如李宏毅老师所指出的，LLM 的规划能力"介于有和无之间"。在简单场景下 LLM 展现出了一定的规划能力，但在复杂约束条件下仍然存在明显局限：

  • PlanBench 的实验结果表明，LLM 可以处理简单的序列规划任务，但在涉及复杂约束条件（如资源限制、时序依赖、互斥条件）时表现显著下降。
  • TravelPlanner 等测试显示，多约束规划（如同时考虑预算、时间、偏好、距离等多个维度）对 LLM 来说仍然极具挑战性。
  • LLM 的规划本质上是"基于模式匹配的推理"而非"基于搜索的最优规划"，在需要回溯和全局优化的场景下容易出错。

改进方向：结合树搜索（Tree Search）算法增强探索能力、使用专门的推理模型（如 o1、DeepSeek-R1）、集成外部规划器（如 PDDL Solver）进行形式化规划，以及通过 Reflexion 等自我纠错机制提升规划的鲁棒性。

面试要点：引用具体的 Benchmark 结果来支撑论点，展现对领域前沿的了解深度。
